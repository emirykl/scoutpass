# Project: ScoutPass

Build a working MVP for **ScoutPass**, a privacy-first football scouting and tryout platform that connects amateur football players with professional clubs and scouts.

ScoutPass allows amateur players to create an AI-generated scouting profile on their own device, selectively share that profile directly with a scout through a peer-to-peer connection, receive a tryout invitation, and optionally receive USD₮ travel support through a self-custodial wallet.

The project must use all three Tether Developers Cup technologies in a meaningful and connected workflow:

- **QVAC** for local AI analysis
- **Pears Stack** for peer-to-peer profile sharing
- **WDK** for self-custodial USD₮ payments

The main product flow must be:

Player performance data
        ↓
QVAC generates a private scouting report
        ↓
Player selects which data to share
        ↓
Pears transfers the report directly to the scout
        ↓
Scout sends a tryout invitation
        ↓
WDK sends optional USD₮ travel support to the player

The application must not use cloud AI. Player performance data and AI prompts must remain on the user’s device.

# 1. Product problem

Many talented amateur football players struggle to reach professional scouts.

Current scouting workflows often depend on:

- Social media messages
- Public highlight videos
- Centralized scouting platforms
- Unstructured PDF documents
- Personal connections
- Unverified tryout invitations

Players usually lose control of their data after sharing it. Scouts also receive inconsistent player profiles that are difficult to compare.

ScoutPass solves this by creating a standardized, player-controlled scouting profile.

The player owns the profile, chooses which information to share, and communicates directly with the scout without uploading the profile to a central scouting database.

# 2. Target users

## Player

An amateur or semi-professional football player who wants to present performance information to scouts and clubs.

The player can:

- Create a local profile
- Enter performance data
- Generate an AI scouting report
- Review and edit the report
- Select which report sections are shared
- Connect directly with a scout
- Receive a tryout invitation
- Receive optional USD₮ travel support

## Scout

A scout, coach or club representative who wants to review standardized player profiles.

The scout can:

- Connect to a player through a Pears invite
- Receive selected player information
- Review the AI-generated report
- Add private evaluation notes
- Send a tryout invitation
- Add optional travel support
- Send USD₮ through WDK

# 5. QVAC integration

Use the official QVAC JavaScript or TypeScript SDK.

Do not use:

- OpenAI API
- Anthropic API
- Gemini API
- Hosted inference APIs
- Any fallback cloud AI service

The QVAC integration must transform the player’s structured statistics and written notes into a standardized scouting report.

## Expected report structure

interface ScoutReport {
  playerSummary: string;
  positionalProfile: string;
  strengths: ReportItem[];
  developmentAreas: ReportItem[];
  playingStyle: string;
  suitableSystems: string[];
  scoutQuestions: string[];
  dataLimitations: string[];
  generatedAt: string;
  modelInfo: string;
}

interface ReportItem {
  title: string;
  explanation: string;
  evidence: string[];
  confidence: "low" | "medium" | "high";
}

## QVAC responsibilities

QVAC should:

- Summarize the player profile
- Interpret statistics according to position
- Combine statistics with coach notes
- Identify potential strengths
- Identify possible development areas
- Describe the player’s playing style
- Suggest tactical systems that may fit the player
- Generate questions a scout should ask
- Explain the limitations of the available data

QVAC must not:

- Claim the player will become professional
- Produce medical advice
- Diagnose injury risk
- Present self-entered data as verified
- Make final recruitment decisions
- Automatically rank players by human value

## Required report disclaimer

This report is generated from information provided by the player. It is intended to support human scouting and does not verify performance claims or replace professional evaluation.

## Reliability requirement

Ask QVAC to return valid structured JSON.

Validate the output with a schema such as Zod.

When the generated response is invalid:

- Retry local generation once
- Show a clear error if it still fails
- Never fabricate a completed report

Example:

interface SharedPlayerPackage {

# 7. Pears Stack integration

Use Pears Stack for all networking.

Do not use:

- Firebase
- Supabase
- Socket.io
- WebSocket backend
- Plain WebRTC
- Central REST backend for player-scout communication

## Pears responsibilities

Use Pears for:

- Peer discovery
- Secure player-scout connection
- Sharing the selected player package
- Sending tryout invitations
- Sending invitation acceptance or rejection
- Synchronizing message status
- Maintaining a local communication history

Recommended building blocks:

- Hyperswarm for peer discovery and connection
- Hypercore for append-only event records
- Corestore for local core management
- Autobase only when multi-writer synchronization is genuinely needed

Do not use Pears merely for a chat box.

The primary P2P object is the scouting relationship and its event history.

## P2P event model

type ScoutPassEvent =
  | PlayerProfileSharedEvent
  | ProfileReceivedEvent
  | TryoutInvitationEvent
  | InvitationResponseEvent
  | TravelSupportProposedEvent
  | TravelSupportSentEvent;

interface BaseEvent {
  id: string;
  type: string;
  senderPublicKey: string;
  createdAt: string;
  protocolVersion: string;
}

## Connection flow

1. Scout selects “Create scouting connection.”
2. The scout application creates an invite code or connection key.
3. The player enters or scans the invite.
4. Both peers connect through Hyperswarm.
5. The player previews the selected report package.
6. The player explicitly approves sharing.
7. The package transfers directly to the scout.
8. The scout stores the received package locally.
9. The scout sends a tryout invitation through the same P2P relationship.

## Privacy requirements

- Do not publish profiles to a public swarm.
- Use a unique topic or invitation per player-scout relationship.
- Validate all incoming messages.
- Add payload size limits.
- Reject unknown event types.
- Do not execute code received from peers.
- Do not expose private keys in logs.
- Store data locally.

# 9. WDK integration

Use Tether WDK for self-custodial wallet functionality.

## WDK responsibilities

The scout should be able to:

- Create or initialize a self-custodial wallet
- View the wallet address
- View test balance
- Enter a travel-support amount
- Review transaction details
- Confirm and sign the payment
- Send supported test USD₮ to the player
- Display transaction status and identifier

The player should be able to:

- Create or initialize a self-custodial wallet
- Share a receive address
- View the incoming test payment
- Associate the payment with the tryout invitation

Use a supported WDK test network and test assets.

Do not use real money for the hackathon demo.

Display a visible “Testnet only” label throughout the payment flow.

## Important wallet rules

- Never log seed phrases or private keys.
- Never commit wallet secrets.
- Never send private keys through Pears.
- Never store secrets in plain-text application state.
- Never automatically send a transaction.
- Require explicit user confirmation.
- Display network, destination, asset and amount before signing.
- Do not claim the payment is escrowed.
- Do not claim that payment verifies the scout or club.

For the MVP, the payment is an optional travel grant sent after the player accepts the invitation.

It is not:

- A tryout fee
- A gambling payment
- A recruitment guarantee
- A legally binding contract
- An escrow arrangement

# 10. Connected three-stack workflow

The three integrations must form one coherent product flow.

Correct integration:

QVAC
Generates the private and standardized player report.

Pears
Lets the player share selected report sections directly with a scout.

WDK
Lets the scout send optional self-custodial travel support after issuing a tryout invitation.

Incorrect integration:

Do not create three unrelated tabs where:

- QVAC is a generic chatbot
- Pears is an empty chat room
- WDK is a disconnected wallet page

Every integration must be visible in the same user journey.

# 15. Security requirements

Create a basic threat model.

Consider:

- Malicious P2P payloads
- Oversized messages
- Fake scout identities
- Self-reported player statistics
- Accidental oversharing
- Wallet secret exposure
- Replay of invitation events
- Duplicate payment attempts
- Corrupted local data

Mitigations:

- Runtime validation with Zod
- Event IDs and deduplication
- Protocol version field
- Payload limits
- Explicit sharing confirmation
- Local-only AI
- No automatic transactions
- Testnet-only payments
- Clear unverified-data warning
- Sanitized logs
- No secrets in source control

Do not claim that ScoutPass fully verifies clubs, scouts or player performance in the MVP.

3. The report passes schema validation.
4. The player selects report sections.
5. A scout creates a Pears invite.
6. The player connects using the invite.
7. The selected report transfers to the scout.
8. The scout views the received report.
9. The scout sends a tryout invitation.
10. The player accepts the invitation.
11. The scout reviews a test USD₮ payment.
12. The scout explicitly signs the WDK transaction.
13. The player sees the transaction status.
14. The application displays no cloud AI dependency.

# 22. Development order

Follow this order strictly:

Step 1:
Create the domain models, forms and local storage.

Step 2:
Implement QVAC local report generation with schema validation.

Step 3:
Build Pears connection and transfer a simple validated test object between two app instances.

Step 4:
Transfer the real selected scouting package.

Step 5:
Implement tryout invitations through Pears.

Step 6:
Integrate WDK wallet initialization and test balance display.

Step 7:
Implement reviewed and user-confirmed test USD₮ payment.

Step 8:
Connect all screens into one complete demo flow.

Step 9:
Add tests, error handling and security checks.

Step 10:
Finish README, architecture documentation and demo script.

Do not prioritize visual polish until QVAC, Pears and WDK work end to end.

Before using SDK methods, inspect the currently installed versions and official documentation. Do not invent package names, functions or transaction methods. Isolate each integration behind an adapter interface so SDK changes can be handled without changing the rest of the application.
